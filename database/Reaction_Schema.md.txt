# Reaction Entity — Detailed Database Schema

This expands the `reactions` entity from the overall chemistry knowledge structure into a full schema covering every field you listed. The core design principle: **fields that are naturally "one value" stay as columns on `reactions`; fields that are naturally "a list of related things" (reactants, keywords, mistakes, references, mechanism steps) become their own tables.** Cramming lists into single text columns is what makes reaction databases painful to search and query later — so it's avoided throughout.

---

## 1. Field-by-Field Modeling Decisions

| Your Field | Why it's modeled this way |
|---|---|
| Reaction ID | Primary key, surrogate `id` (UUID or BIGSERIAL) |
| Reactants / Products | **Not** two separate lists — one polymorphic junction table `reaction_participants` with a `role` column. A species can be an element or a compound, so it references *either* `element_id` or `compound_id` (never both). This also naturally holds stoichiometric coefficients and physical state (s/l/g/aq). |
| Reaction Type | Lookup table `reaction_types`, many-to-many — a reaction is often more than one type at once (e.g., a reaction can be both *redox* and *combustion*, or both *acid-base* and *precipitation*) |
| Balanced Equation | Stored as a display string + LaTeX on `reactions`, but *derivable* from `reaction_participants` — kept redundantly for fast rendering, generated from participants at write time |
| Conditions | Free-form conditions beyond catalyst/temp/pressure (light, pH, solvent, inert atmosphere) don't fit fixed columns — stored as JSONB `conditions` |
| Catalyst | Own junction table `reaction_catalysts` (a reaction can have zero, one, or several catalysts, and a catalyst is itself a compound/element so it should be a real reference, not free text) |
| Temperature / Pressure | Modeled as ranges (`min`, `max`, `unit`) since most reactions specify optimal ranges, not single points |
| Observation | Text column — what a student visually/physically observes (color change, gas evolved, precipitate) |
| Mechanism | Structured as an **ordered steps table** `reaction_mechanism_steps`, not one long text block — because mechanisms are inherently sequential and the app can animate/step through them |
| Explanation | Long-form text/markdown column on `reactions` — the pedagogical write-up |
| Level / Chapter | Junction table `reaction_curriculum` — the same reaction (e.g., esterification) legitimately appears in multiple chapters across HSC and Honours, so this must be many-to-many, with an `is_primary` flag for the "default" placement |
| Keywords | Tag table `keywords` + junction `reaction_keywords` — enables autocomplete/search and avoids duplicate keyword strings across reactions |
| Common Mistakes | Own table `reaction_common_mistakes` — each mistake is really a (misconception, correction) pair, better as structured rows than a flat paragraph |
| References | Own table `references` + junction `reaction_references` — sources get reused across many reactions (a textbook cited 200 times shouldn't be retyped 200 times) |

## 2. Full SQL Schema

```sql
-- ============================================================
-- CORE REACTION TABLE
-- ============================================================
CREATE TABLE reactions (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name                TEXT NOT NULL,                     -- e.g. "Esterification of Ethanoic Acid"
    balanced_equation    TEXT NOT NULL,                     -- e.g. "CH3COOH + C2H5OH ⇌ CH3COOC2H5 + H2O"
    balanced_equation_latex TEXT,                           -- LaTeX render string
    conditions          JSONB,                              -- {"light": "UV", "solvent": "none", "pH": "acidic"}
    observation         TEXT,                                -- what the student sees happen
    mechanism_summary   TEXT,                                -- short overview (detail lives in reaction_mechanism_steps)
    explanation         TEXT,                                -- full pedagogical write-up (markdown)
    temp_min            NUMERIC,
    temp_max            NUMERIC,
    temp_unit           TEXT DEFAULT 'C',
    pressure_min        NUMERIC,
    pressure_max        NUMERIC,
    pressure_unit       TEXT DEFAULT 'atm',
    is_reversible       BOOLEAN DEFAULT FALSE,
    is_exothermic       BOOLEAN,
    created_at          TIMESTAMPTZ DEFAULT now(),
    updated_at          TIMESTAMPTZ DEFAULT now(),
    content_version     INT DEFAULT 1                        -- for incremental mobile sync
);

-- ============================================================
-- REACTION TYPE (lookup, many-to-many)
-- ============================================================
CREATE TABLE reaction_types (
    id      SERIAL PRIMARY KEY,
    name    TEXT UNIQUE NOT NULL,          -- 'Redox', 'Acid-Base', 'Addition', 'Substitution',
                                            -- 'Elimination', 'Precipitation', 'Combustion', 'Polymerization'...
    category TEXT                          -- broader grouping, e.g. 'Organic', 'Inorganic', 'Physical'
);

CREATE TABLE reaction_reaction_types (
    reaction_id      UUID REFERENCES reactions(id) ON DELETE CASCADE,
    reaction_type_id INT REFERENCES reaction_types(id) ON DELETE RESTRICT,
    PRIMARY KEY (reaction_id, reaction_type_id)
);

-- ============================================================
-- REACTANTS & PRODUCTS (polymorphic: element OR compound)
-- ============================================================
CREATE TABLE reaction_participants (
    id                          BIGSERIAL PRIMARY KEY,
    reaction_id                 UUID REFERENCES reactions(id) ON DELETE CASCADE,
    role                        TEXT NOT NULL CHECK (role IN ('reactant','product')),
    compound_id                 UUID REFERENCES compounds(id),
    element_id                  UUID REFERENCES elements(id),
    stoichiometric_coefficient  INT NOT NULL DEFAULT 1,
    physical_state              TEXT CHECK (physical_state IN ('s','l','g','aq')),
    sequence_order              INT DEFAULT 0,               -- preserves left-to-right equation order
    CONSTRAINT one_species_only CHECK (
        (compound_id IS NOT NULL AND element_id IS NULL) OR
        (compound_id IS NULL AND element_id IS NOT NULL)
    )
);

CREATE INDEX idx_participants_reaction ON reaction_participants(reaction_id);
CREATE INDEX idx_participants_compound ON reaction_participants(compound_id);
CREATE INDEX idx_participants_element  ON reaction_participants(element_id);

-- ============================================================
-- CATALYSTS (own table — a reaction may have 0, 1, or several)
-- ============================================================
CREATE TABLE reaction_catalysts (
    reaction_id  UUID REFERENCES reactions(id) ON DELETE CASCADE,
    compound_id  UUID REFERENCES compounds(id),
    element_id   UUID REFERENCES elements(id),
    note         TEXT,                                       -- e.g. "finely divided, acts as surface catalyst"
    CONSTRAINT one_species_only_cat CHECK (
        (compound_id IS NOT NULL AND element_id IS NULL) OR
        (compound_id IS NULL AND element_id IS NOT NULL)
    )
);

-- ============================================================
-- MECHANISM (ordered steps, not a single blob)
-- ============================================================
CREATE TABLE reaction_mechanism_steps (
    id            BIGSERIAL PRIMARY KEY,
    reaction_id   UUID REFERENCES reactions(id) ON DELETE CASCADE,
    step_number   INT NOT NULL,
    description   TEXT NOT NULL,
    diagram_url   TEXT,                                       -- arrow-pushing diagram, if available
    UNIQUE (reaction_id, step_number)
);

-- ============================================================
-- CURRICULUM PLACEMENT (Level / Chapter — many-to-many)
-- ============================================================
CREATE TABLE reaction_curriculum (
    reaction_id  UUID REFERENCES reactions(id) ON DELETE CASCADE,
    chapter_id   UUID REFERENCES chapters(id) ON DELETE CASCADE,
    is_primary   BOOLEAN DEFAULT FALSE,       -- the "home" chapter for default navigation
    PRIMARY KEY (reaction_id, chapter_id)
);
-- Level is reached via chapters.subject_id -> subjects.level_id (no need to duplicate level_id here)

-- ============================================================
-- KEYWORDS / TAGS
-- ============================================================
CREATE TABLE keywords (
    id    SERIAL PRIMARY KEY,
    term  TEXT UNIQUE NOT NULL
);

CREATE TABLE reaction_keywords (
    reaction_id UUID REFERENCES reactions(id) ON DELETE CASCADE,
    keyword_id  INT REFERENCES keywords(id) ON DELETE CASCADE,
    PRIMARY KEY (reaction_id, keyword_id)
);

-- ============================================================
-- COMMON MISTAKES
-- ============================================================
CREATE TABLE reaction_common_mistakes (
    id              BIGSERIAL PRIMARY KEY,
    reaction_id     UUID REFERENCES reactions(id) ON DELETE CASCADE,
    mistake_text    TEXT NOT NULL,             -- the misconception itself
    correction_text TEXT NOT NULL,             -- what's actually correct
    sequence_order  INT DEFAULT 0
);

-- ============================================================
-- REFERENCES (reusable citation pool)
-- ============================================================
CREATE TABLE "references" (
    id           SERIAL PRIMARY KEY,
    citation     TEXT NOT NULL,                -- e.g. "NCTB HSC Chemistry 2nd Paper, Ch. 3"
    source_type  TEXT,                          -- 'textbook','journal','website'
    url          TEXT
);

CREATE TABLE reaction_references (
    reaction_id   UUID REFERENCES reactions(id) ON DELETE CASCADE,
    reference_id  INT REFERENCES "references"(id) ON DELETE CASCADE,
    page_or_section TEXT,
    PRIMARY KEY (reaction_id, reference_id)
);
```

## 3. Entity-Relationship Summary

```
reactions (1) ───< reaction_participants >─── (1) compounds / elements   [role: reactant | product]
reactions (1) ───< reaction_catalysts >──── (1) compounds / elements
reactions (1) ───< reaction_mechanism_steps                              [ordered]
reactions (M) ──── reaction_reaction_types ────< reaction_types (M)
reactions (M) ──── reaction_curriculum ────< chapters (M) ──── subjects ──── levels
reactions (M) ──── reaction_keywords ────< keywords (M)
reactions (1) ───< reaction_common_mistakes                              [ordered]
reactions (M) ──── reaction_references ────< references (M)
```

Everything hangs off `reactions.id`. Nothing about a reaction is stored anywhere else — this is the single canonical record queried by concepts, experiments, and the curriculum tree alike (via `concept_reactions` and `experiment_reactions` from the broader schema).

## 5. Sample Query: "Give me everything about a reaction, assembled"

```sql
SELECT
    r.name, r.balanced_equation, r.observation, r.explanation,
    r.temp_min, r.temp_max, r.temp_unit,
    json_agg(DISTINCT rt.name)              AS types,
    json_agg(DISTINCT jsonb_build_object(
        'role', rp.role, 'coefficient', rp.stoichiometric_coefficient
    ))                                        AS participants,
    json_agg(DISTINCT k.term)                 AS keywords
FROM reactions r
LEFT JOIN reaction_reaction_types rrt ON rrt.reaction_id = r.id
LEFT JOIN reaction_types rt          ON rt.id = rrt.reaction_type_id
LEFT JOIN reaction_participants rp   ON rp.reaction_id = r.id
LEFT JOIN reaction_keywords rk       ON rk.reaction_id = r.id
LEFT JOIN keywords k                 ON k.id = rk.keyword_id
WHERE r.id = '11111111-1111-1111-1111-111111111111'
GROUP BY r.id;
```

In practice, the app backend would run a handful of targeted queries (participants, mechanism steps, mistakes, references separately) rather than one giant aggregated join — cleaner to map into JSON for the mobile client and avoids the row-multiplication problems that come from joining several one-to-many tables at once.

## 6. Why Not Just Put Everything in One Wide Table with Text/Array Columns?

It's tempting to use a single `reactions` table with `reactants TEXT[]`, `keywords TEXT[]`, `common_mistakes TEXT[]`, etc. This works for a v1 prototype but breaks down for a real app:

- **No referential integrity** — a reactant stored as `"H2SO4"` text can't be validated against the actual compound record, and a typo silently creates an orphaned string.
- **Can't query "which reactions use H2SO4 as a catalyst"** without string matching across every row.
- **No stoichiometric coefficients or physical state** attachable to a plain string array element.
- **Keyword search/autocomplete** across reactions needs a real keyword table to avoid "ester" vs "esterification" vs "Esterification" being treated as different tags.

The junction-table approach costs a bit more upfront schema design but pays off the moment the app needs cross-cutting features: search, "related reactions," quiz generation from common mistakes, or citation management.

---

## 7. Indexing & Performance Notes

- `reaction_participants(compound_id)` and `(element_id)` — enables fast "which reactions is this compound in" queries.
- `reaction_curriculum(chapter_id)` — enables fast "all reactions in this chapter" for the navigation view.
- GIN index on `reactions.conditions` (JSONB) if the app ever needs to filter by condition keys.
- Full-text search index (`tsvector`) on `reactions.explanation` and `mechanism_summary` for the app's global search bar.
- `content_version` + `updated_at` on `reactions` (and ideally on every child table) to support incremental sync to the mobile SQLite mirror — only pull rows changed since last sync.
