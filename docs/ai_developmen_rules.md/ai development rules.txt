Chemistry AI — 20 Development Rules for AI-Assisted Software Development
Folder Naming (Rules 1–3)
1. Use lowercase kebab-case for all folders and files (e.g. reaction-engine, quiz-service, not ReactionEngine or Quiz_Service). This avoids cross-platform casing bugs (macOS/Windows are case-insensitive, Linux isn't) and keeps things consistent whether a human or an AI coding assistant generates the file.
2. Mirror the domain, not the framework, in top-level folder names. Structure by feature/domain (/reaction-prediction, /molecular-viewer, /quiz-engine) rather than by technical layer (/controllers, /services, /utils) at the top level. This keeps AI-generated code scoped to the right domain instead of scattering related logic across generic buckets.
3. Every AI-generated module gets its own folder with a co-located README.md. When an AI assistant scaffolds a new service or component, it must create a short README in that folder explaining purpose and inputs/outputs — this prevents "orphaned" AI-generated code that nobody, human or AI, can later explain.
Coding Style (Rules 4–6)
4. Enforce a single formatter and linter per language, run automatically on save and in CI (Prettier + ESLint for TypeScript, Black + Ruff for Python). AI-generated code must pass the same linter as human-written code — no exceptions, no "AI code is exempt from style checks."
5. No AI-generated code merges without explicit type annotations (TypeScript types, Python type hints). Chemistry data structures (molecules, reactions, quiz payloads) are easy to get subtly wrong; types catch mismatches that a fast-moving AI-assisted PR review might miss.
6. Prefer explicit, verbose function and variable names over clever abbreviations, especially in chemistry logic (limitingReagentMoles not lrMol). This matters more in AI-assisted codebases than usual, since future prompts to an AI assistant will rely on the code being self-explanatory context, not tribal knowledge.
Git Commit (Rules 7–9)
7. Use Conventional Commits format (feat:, fix:, docs:, refactor:, test:, chore:) for every commit, including AI-generated ones. This keeps changelogs and release notes auto-generatable and makes it easy to audit which commits came from AI-assisted sessions versus manual work.
8. Every commit that includes AI-generated code must note it in the commit body (e.g. AI-assisted: reaction prediction logic drafted with Claude, reviewed by <name>). This isn't about distrust — it's about traceability, so if a bug surfaces later, the team knows which parts had less direct human authorship and may need closer review.
9. No direct commits to main/develop — all changes, including small AI-suggested fixes, go through a pull request with at least one human reviewer. AI assistants can draft PRs, but merging is always a human decision, especially in a domain (chemistry) where a subtly wrong constant or formula can silently produce incorrect answers.
Documentation (Rules 10–12)
10. Every public API endpoint and AI-facing function must have a docstring stating inputs, outputs, and known limitations (e.g. "reaction prediction confidence drops for uncommon organometallic reactions"). This is especially important for functions the AI tutoring layer calls, since the LLM needs to know when to hedge its explanations to students.
11. Maintain a single /docs/architecture.md, kept current with every major structural change, rather than letting documentation live only in scattered PR descriptions or Slack threads. AI assistants asked to make future changes should be able to read this one file and get accurate context, instead of guessing from stale comments.
12. Document every prompt template used in production (the system prompts and few-shot examples driving the AI tutor, quiz generator, and reaction predictor) in a version-controlled /prompts/ directory, not hardcoded inline strings buried in service code. Prompts are a core part of the system's behavior and deserve the same change-tracking as code.
Testing (Rules 13–15)
13. Every deterministic chemistry function (equation balancer, stoichiometry calculator, molecular property calculator) requires unit tests with known correct answers before merge — no exceptions for AI-generated implementations. Correctness here is non-negotiable; test against textbook examples, not just "it looks right."
14. AI-generated or AI-assisted code requires the same test coverage threshold as human-written code (e.g. minimum 80% for core services) — CI should not distinguish origin when enforcing coverage gates.
15. Maintain a regression test set of "known-tricky" chemistry queries (edge cases like redox balancing, organic mechanisms with multiple valid products) that every model or prompt update must be re-run against before deployment, to catch silent quality regressions when the AI tutoring layer changes.
Prompt Engineering (Rules 16–18)
16. Every production prompt must explicitly state the model's role, audience level, and failure mode (e.g. "You are a chemistry tutor for high-school students; if uncertain about a reaction outcome, say so rather than guessing"). Ambiguous prompts are the leading cause of confidently wrong chemistry explanations.
17. Separate deterministic computation from generative explanation at the prompt level — never ask the LLM to both calculate a numeric answer (like molar mass) and explain it in the same free-form generation step. Compute with RDKit/deterministic code, then hand the verified result to the LLM purely for explanation. This is the single most important rule for keeping factual accuracy high.
18. Version and A/B test prompt changes like code changes — no prompt goes to production without being tested against the regression set (Rule 15) and reviewed by at least one other engineer, since a "small wording tweak" can measurably change explanation quality or introduce inaccuracies.
Security (Rules 19–20)
19. Never pass raw user input directly into a prompt without sanitization, and never let the LLM's output be executed as code or used to construct database queries directly — treat all model output as untrusted input, the same way you'd treat any user-submitted data, to prevent prompt injection or downstream injection attacks.
20. Any AI-assisted code touching authentication, payment, or user data storage requires mandatory manual security review before merge, regardless of how confident the AI assistant's output appears — AI-generated code is held to a stricter review bar in these areas, not a looser one, since these are exactly the places where a plausible-looking but subtly flawed implementation causes the most damage.
